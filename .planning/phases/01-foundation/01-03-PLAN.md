# Plan 01-03: QR Code Generation & Firewall

<context>
Once the LAN IP is established, the server must display a QR code locally and warn users if Windows Firewall is potentially blocking the connection. This satisfies the primary usability goal: users must simply point their phone camera at the screen and scan the QR code to connect and initiate the Web Data Transfer project.
</context>

<requirements>
- [ ] Use `qrcode-terminal` to generate an ASCII QR code in the terminal.
- [ ] Render the payload `http://<lanIp>:<port>`.
- [ ] On startup (Windows only), output a firewall warning message outlining the command required if device connection fails.
- [ ] Ensure terminal display order is: Started Message -> Windows Firewall Hint -> Huge QR Code.
</requirements>

<must_haves>
- An unmistakable QR Code generated in the terminal with its encoded string clearly being the dynamically calculated server IP address.
- Windows Firewall hint command dynamically populated with the exact `<PORT>`.
</must_haves>

<tasks>
<task>
Install `qrcode-terminal` running `npm install qrcode-terminal`. Require it inside `server.js`.
</task>
<task>
Call `qrcode.generate("http://${lanIp}:${port}", {small: true})` in `server.js` immediately after starting the server string formatting message. This prints an ASCII code.
</task>
<task>
Determine if platform is Windows using `os.platform() === 'win32'`. If yes, console.log a warning line with exact instructions indicating inbound blocking might occur. Print the required command: "netsh advfirewall firewall add rule name=\"Web Data Transfer\" dir=in action=allow protocol=TCP localport=" + port.
</task>
</tasks>

<verification>
1. Run `npm start`.
2. Observe terminal displaying firewall connection instructions.
3. QR code rendered below the instructions in terminal format. Scanning it with a phone returns `http://<LAN_IP>:<PORT>`.
</verification>
