# Plan 06-02: Cross-Browser Validation and Windows Firewall (COMPAT-02)

## Context
Ensure the application correctly spins up on a new PC across Edge, Firefox, and Chrome. The current server successfully configures a Local IP discovery engine and prints a valid Administrator `netsh` Firewall override rule directly to the user's terminal to ensure connections span to mobile devices seamlessly.

## Implementation Details
1. **Firewall Evaluation**:
   - In `server.js`, we have implemented `console.log` instructions outputting: `netsh advfirewall firewall add rule name="Web Data Transfer" dir=in action=allow protocol=TCP localport=3000`.
   - Ensure the server script clearly outputs instructions when starting up so the end user can execute this if their phone fails to reach the site.
2. **PC Cross-Browser**:
   - Provide fallback logic or warnings if `window.showDirectoryPicker` or `window.showSaveFilePicker` (HTML5 File System API) is completely unsupported or restricted by certain Firefox versions.
   - We must provide a conditional fallback if `!window.showDirectoryPicker` natively exists on older instances to guide the user towards standard `<input type="file" multiple>` or a visual warning. This happens currently with an `alert()` on the HTML UI, but we must make sure all drag-and-drop (`ondrop` event loop parsing) covers File objects securely across the matrix.

## Success Criteria
- [ ] If Firefox restricts full FileSystem API access, the explicit alerts correctly manage expectations and fallback to standard Drag-and-Drop behaviors smoothly without uncaught Promise rejections natively.
- [ ] Drag-and-drop `DataTransferItemList` parsing acts identically across `Edge/Chrome/Firefox` natively executing WebRTC DataChannels properly on the backend.
