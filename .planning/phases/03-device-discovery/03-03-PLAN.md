# Plan 03-03: Auto-Discovery for Returning Devices

<context>
Using `localStorage` device tracking established in 03-02, returning devices need to bypass the QR flow and seamlessly merge directly back into the signaling server's "room" logic relying purely on the internal tracking of `socket.handshake.address`.
</context>

<requirements>
- [ ] Connect existing stored device IDs directly to `socket.io` immediately upon page load.
- [ ] The PC/Host broadcasts actively connected sub-devices automatically to all joining users the moment a new socket hits the `remoteFamily` or LAN IP room.
- [ ] Render visual feedback: Ensure the PC accurately counts the amount of discovered external users on the local net without manually rescanning codes.
</requirements>

<must_haves>
- Eliminates the need for re-scanning codes or checking `?pair=` queries iteratively for subsequent visits on identical LAN hubs.
- Automatically pushes down updated connected lists to devices the second a user navigates onto `192.168.x.x:3000`.
</must_haves>

<tasks>
<task>
In `server.js`, expand the `io.on('connection')` block to emit a `request_sync` command to all other devices in the same IP room as soon as it binds locally.
</task>
<task>
In `public/index.html`, add a function binding to `socket.on('request_sync')`. When the host PC receives this, it should reply emitting `sync_response` containing its own cached data, broadcasting to all peers so the newly connected device retrieves an up-to-date peer list seamlessly.
</task>
<task>
Add logic checking `localStorage.getItem('deviceId')`. If it exists upon standard HTTP domain loading (no `?pair=` query string required), establish a fully fledged, fully trusted WebSocket signaling room.
</task>
</tasks>
