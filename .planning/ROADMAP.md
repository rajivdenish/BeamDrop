# Roadmap: Web Data Transfer

## Overview

Six phases built the LAN file transfer tool from the ground up, with a newly added Phase 7 for VPS deployment. Phase 1 established the server foundation. Phase 2 added invite-only authentication. Phase 3 implemented device discovery. Phase 4 delivered bidirectional file transfer via WebRTC DataChannel. Phase 5 added text and clipboard transfer. Phase 6 validated cross-browser and platform operation. Phase 7 implements standalone executable bundling and containerized deployment for VPS environments.

## Phases

## Phase Details

### Phase 1: Foundation
**Goal**: A phone on the same WiFi can scan a QR code and load the app in its browser — proven on real hardware with no configuration
**Depends on**: Nothing (first phase)
**Requirements**: (none directly — establishes infrastructure all other requirements depend on)
**Success Criteria** (what must be TRUE):
  1. Running `node server.js` (or equivalent) starts the server and prints the LAN URL and QR code to the terminal within 3 seconds
  2. A phone on the same WiFi network opens the app by scanning the printed QR code — no manual IP entry required
  3. The QR code encodes the correct LAN IP (not 127.0.0.1, not a Docker bridge, not a VPN adapter) on a multi-NIC machine
  4. The server starts and is reachable on a fresh Windows machine where Windows Firewall is the only barrier — startup output includes the exact firewall command needed if inbound connections are blocked

### Phase 2: Authentication
**Goal**: Only invited users can access the tool — users can register, log in, stay logged in, and log out securely
**Depends on**: Phase 1
**Requirements**: AUTH-01, AUTH-02, AUTH-03, AUTH-04, AUTH-05
**Success Criteria** (what must be TRUE):
  1. A new user can register with email and password using an invite link generated by the admin — registration is rejected without a valid invite link
  2. An admin can generate invite links from an admin panel and manually approve pending registrations
  3. A logged-in user stays logged in across browser sessions (closing and reopening the browser tab does not force re-login)
  4. A user can log out from any page and is immediately redirected to the login screen with the session invalidated

### Phase 3: Device Discovery
**Goal**: Devices find each other on the LAN — new devices pair via QR code, returning devices reappear automatically
**Depends on**: Phase 2
**Requirements**: DISC-01, DISC-02, DISC-03
**Success Criteria** (what must be TRUE):
  1. A phone that has never connected before scans the QR code displayed in the PC browser and appears as an available device in the UI within 5 seconds
  2. A phone that has connected before opens the app URL directly (without scanning QR again) and appears automatically in the device list on the same LAN session
  3. Devices connected from a different network (different public IP) are not visible to the current user — each user sees only devices on their current LAN segment

### Phase 4: File Transfer
**Goal**: Files move between any two discovered devices over a direct P2P connection — phone to PC and PC to phone, with visible progress
**Depends on**: Phase 3
**Requirements**: FILE-01, FILE-02, FILE-03, FILE-04, FILE-05, FILE-06
**Success Criteria** (what must be TRUE):
  1. A user selects one or more files on their phone using a file picker and sends them — the files appear in the PC's receive folder and the server never stores the file data
  2. A user drags and drops one or more files onto the browser UI and sends them in a single operation
  3. Files queued on the PC are browseable on the phone browser and can be downloaded with a tap
  4. A real-time progress bar is visible on both the sending and receiving device during every transfer
  5. File data travels over a WebRTC DataChannel between the two peers — verified by confirming the server receives no file bytes during transfer

### Phase 5: Text and Clipboard
**Goal**: Users can transfer text and clipboard content between devices instantly without files
**Depends on**: Phase 4
**Requirements**: TEXT-01, TEXT-02
**Success Criteria** (what must be TRUE):
  1. A user types or pastes text on either device and sends it — the text appears on the receiving device within 2 seconds over the existing P2P connection
  2. Pressing Ctrl+V on the PC with clipboard content sends that content to the connected phone instantly — no extra button clicks required
  3. On devices where the Clipboard API is unavailable (HTTP context on mobile), received text is displayed in a pre-selected text area so the user can copy it manually

### Phase 6: Compatibility and Polish
**Goal**: The complete feature set works reliably across all target browsers and platforms — mobile and desktop
**Depends on**: Phase 5
**Requirements**: COMPAT-01, COMPAT-02
**Success Criteria** (what must be TRUE):
  1. All transfer features (file send, file receive, text, clipboard) work on Chrome for Android and Safari for iOS without any browser extensions or configuration changes
  2. All transfer features work on the PC in Chrome, Firefox, and Edge
  3. iOS Safari-specific behaviors are handled correctly — files download rather than opening inline, background tab suspension does not silently kill an active transfer
  4. The app functions correctly on a fresh Windows machine with default security settings after following the startup firewall instructions printed by the server

### Phase 7: Deployment
**Goal**: The complete application can be distributed as a standalone portable .exe and securely hosted on a public VPS as a Docker container.
**Depends on**: Phase 6
**Requirements**: DEPLOY-01, DEPLOY-02
**Success Criteria** (what must be TRUE):
  1. A Dockerfile exists to compile the Node server and dependencies into a lightweight image.
  2. A docker-compose.yml file seamlessly mounts volume persistency locally to save database auth state over resets.
  3. The `server.js` file can run indefinitely on a VPS mapping public port 3000 effectively.
